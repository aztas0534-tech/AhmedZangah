# معيار المستندات (Best Practices) — AZTA ERP

هذا المستند يحدد “الحد الأدنى الإلزامي” لمحتوى المستندات المطبوعة/المؤرشفة داخل النظام، بهدف حماية الحقوق، تسهيل التدقيق، وتوحيد شكل المستندات.

## مبادئ عامة (تنطبق على كل المستندات)

### 1) التعريف والمرجعية
- رقم مستند واضح (Human-readable) + المعرّف الداخلي (UUID) أو جزء منه.
- نوع المستند (فاتورة/سند قبض/سند صرف/مرتجع/قيد يومية/استلام/تحويل…).
- الحالة (Draft / Posted / Voided / Approved…) مع تاريخها إن توفر.
- مرجع المصدر: جدول المصدر + معرف المصدر (مثال: orders / payments / purchase_receipts).

### 2) الأطراف (Counterparty)
- اسم الطرف (عميل/مورد/طرف مالي).
- رقم هاتف/عنوان إن توفر.
- عند التحصيل/الدفع: “استلمنا من” أو “دُفع إلى” بشكل صريح.

### 3) العملات وسعر الصرف
- عملة المستند (Currency Code).
- إن كانت مختلفة عن العملة الأساسية: سعر الصرف + العملة الأساسية + المبالغ بالعملتين (قدر الإمكان).
- سياسة توضيح: هل سعر الصرف “نظام” أم “يدوي”.

### 4) ملخص مالي
- إجمالي قبل ضريبة/خصم، الضريبة، الخصم، إجمالي نهائي.
- طريقة الدفع + تفاصيل العملية (رقم حوالة/إيداع/شبكة) عند الطرق غير النقدية.
- عند الآجل: شروط الدفع + تاريخ الاستحقاق + رصيد متبقٍ إن توفر.

### 5) التدقيق والأرشفة (Audit & Archiving)
- تاريخ الطباعة.
- طُبع بواسطة (اسم المستخدم/الموظف).
- إن أمكن: تاريخ الإنشاء/الاعتماد/الترحيل ومن قام بها.
- مكان للتوقيعات حسب نوع المستند (Prepared/Checked/Approved/Receiver).
- توثيق “عدد مرات الطباعة” للفواتير (Invoice Print Count) إن كان مطلوبًا.

## معايير حسب نوع المستند

### أ) فاتورة بيع (نقد/آجل)
- رقم الفاتورة + UUID الطلب (أو جزء منه) + تاريخ الإصدار.
- بيانات العميل.
- تفاصيل الأصناف (الصنف/الوحدة/UOM/الكمية/السعر/الإضافات).
- العملة وسعر الصرف عند تعدد العملات.
- طريقة الدفع + تفاصيل العملية عند الحوالات/الإيداع.
- عند الآجل: شروط السداد + تاريخ الاستحقاق + أي مرجع للائتمان.

### ب) سند قبض (Receipt Voucher)
- رقم السند + رقم القيد المحاسبي المرتبط إن وجد.
- “استلمنا من” (الطرف) + رقم العملية + اسم المرسل/الهاتف (عند حوالة/إيداع).
- إلى حساب (Debit) ومن حساب (Credit) كأسماء/أكواد.
- مرجع الدفعة (payment_id) + مرجع الطلب/الفاتورة إن وجد.

### ج) سند صرف (Payment Voucher)
- رقم السند + الطرف (مورد/طرف).
- من حساب (Credit) إلى حساب (Debit) بوضوح.
- رقم العملية/الشيك/التحويل + المستلم + تاريخ الصرف.

### د) قيد يومية (Journal Voucher)
- رقم القيد + نوعه + وصف واضح.
- قائمة القيود (Account Code/Name + Debit/Credit) + العملة.
- سبب القيد/مرجعه (مصدره).

### هـ) مردودات المبيعات (Sales Return Note / Credit Note)
- رقم الإشعار + مرجع الفاتورة/الطلب.
- بيانات العميل + سبب المرتجع.
- تفاصيل الأصناف والكمية بوحدة البيع ووحدة الأساس.
- طريقة الإرجاع (نقد/شبكة/ذمم/رصيد عميل) + إجمالي مسترد + ضريبة مسترجعة.

### و) مردودات المشتريات (Supplier Return Note)
- رقم الإشعار + مرجع أمر الشراء + المورد.
- سبب المرتجع + تفاصيل الأصناف والكميات.
- إجمالي المرتجع بالعملة + بالعملة الأساسية إذا لزم + سعر الصرف.

### ز) سند استلام مخزني (GRN)
- رقم GRN + مرجع أمر الشراء/الاستلام.
- المستودع + المورد + تاريخ الاستلام.
- الأصناف + الدفعات/التواريخ (إنتاج/انتهاء) عند توفرها.
- توقيع أمين المخزن/المستلم.

### ح) تحويل بين المخازن
- رقم التحويل + من/إلى مستودع + تاريخ.
- الأصناف والكميات + ملاحظات.
- توقيع المُرسل/المُستلم.

### ط) كشف حساب طرف مالي
- اسم الطرف + معرفه + الفترة + الحساب/العملة.
- رصيد جاري + إجماليات مدين/دائن.
- تفاصيل الحركات مع المرجع والرصيد التراكمي + حالة التصفية/المطابقة إن وجدت.

## ملاحظة تنفيذية
- معيار المحتوى يُطبق على مستوى “البيانات” أولًا (تخزين metadata داخل payments/orders/accounting_documents) ثم “العرض” داخل القوالب.
- عند اختلاف بيئات قاعدة البيانات (هجرات غير مطبقة/دوال ناقصة): يجب وجود fallbacks في الواجهة + سجلات تدقيق في system_audit_logs لضمان عدم ضياع الأثر.
